Проведи полный security-аудит этого проекта в несколько проходов.

---

---

## ПРОХОД 1 — Картография проекта

Составь полную карту проекта:

- Все endpoints бэкенда и их методы (GET, POST, PUT, DELETE)
- Все места где принимается пользовательский ввод
- Все внешние зависимости и пакеты
- Все файлы конфигурации, .env, секреты
- Все места загрузки файлов
- Все редиректы

Запиши карту в `SECURITY_AUDIT_LOG.md`. Не переходи к Проходу 2, пока карта не готова.

---

## ПРОХОД 2 — Проверка уязвимостей

Проверяй строго по одному пункту за раз, используя карту из Прохода 1. Не переходи к следующему пункту, пока текущий полностью не проверен по всему проекту. После каждого пункта обновляй `SECURITY_AUDIT_LOG.md`.

### 1. XSS (Cross-Site Scripting)

Найди все места где пользовательский ввод:
- Отображается на странице без экранирования
- Вставляется через `innerHTML` или `document.write`
- Попадает в HTML-атрибуты без санитизации
- Передаётся в `eval()` или подобные функции

### 2. Незащищённые API Endpoints

- Найди endpoints доступные без токена или сессии
- Проверь, нельзя ли обычному пользователю вызвать admin-маршруты
- Убедись что аутентификация и авторизация есть на каждом маршруте

### 3. Хранение секретов в коде

- Найди API-ключи, токены, пароли, строки подключения захардкоженные в коде
- Проверь `.env` файлы — не попали ли они в репозиторий
- Проверь git history на случайно закоммиченные секреты

### 4. Небезопасные заголовки HTTP

Проверь, выставлены ли заголовки:
- `Content-Security-Policy`
- `X-Frame-Options`
- `X-Content-Type-Options`
- `Strict-Transport-Security`

### 5. Отсутствие Rate Limiting

Найди endpoints без ограничения количества запросов — особенно:
- `login`, `register`, `password-reset`, `OTP / 2FA`

### 6. Небезопасная загрузка файлов

Если есть upload файлов, проверь:
- Валидируется ли тип файла на сервере (не только на клиенте)
- Есть ли ограничение размера файла
- Сохраняются ли файлы вне публичной папки
- Можно ли загрузить `.exe`, `.php`, `.js` файл

### 7. Неправильная настройка CORS

- Проверь `Access-Control-Allow-Origin: *` в сочетании с куками или токенами
- Убедись что указаны конкретные разрешённые домены, а не wildcard

### 8. Зависимости с известными уязвимостями

- Запусти `npm audit` или аналог для используемого стека
- Найди пакеты с критическими и высокими CVE-уязвимостями
- Предложи обновления или замены

### 9. Утечка информации в ошибках

Найди все места где сервер возвращает:
- Stack trace, имена внутренних функций, пути файловой системы, детали базы данных

В продакшене пользователь должен видеть только общее сообщение об ошибке.

### 10. Небезопасные редиректы

- Найди все редиректы использующие параметры из URL или тела запроса
- Убедись что URL валидируется перед редиректом и нельзя отправить пользователя на внешний сайт

---

## ПРОХОД 3 — Финальный отчёт

1. Перечитай все найденные проблемы
2. Оцени каждую по критичности:

| Уровень | Описание |
|---|---|
| 🔴 Critical | Немедленная угроза, эксплуатируется легко |
| 🟠 High | Серьёзная уязвимость, исправить срочно |
| 🟡 Medium | Умеренный риск, исправить в ближайшем спринте |
| 🟢 Low | Незначительный риск, улучшить по возможности |

3. Сформируй итоговый список в порядке приоритета
4. Для каждой проблемы укажи: файл, строку кода, описание, способ исправления
5. Запиши всё в `SECURITY_AUDIT_LOG.md`

Если по какому-то пункту проблем не найдено — явно напиши "чисто". Не пропускай файлы. Если файл большой — проверь его целиком.
